TALENT ACQUISITION SYSTEM - PROJECT MEETING SUMMARY
============================================

1. PROJECT OVERVIEW
-------------------
This project is a Flask-based Talent Acquisition System web application. It allows users to upload a resume in PDF, DOCX, or TXT format, analyzes the resume against common Talent Acquisition System criteria, compares it with an optional target job role or job description, recommends suitable job roles, and stores scan history for logged-in users.

The main purpose of the project is to help job seekers understand how well their resume performs for TAS screening and how closely it matches a target role. The system gives a total score, category-wise scoring, job matches, missing keywords, improvement suggestions, and suggested bullet rewrites.

The application also includes user registration, login, report history, PDF export, contact form submission, and an admin dashboard for viewing contact messages and managing the job recommendation catalog.


2. TECHNOLOGY STACK
-------------------
Backend framework:
- Flask is used as the web framework.
- Flask handles routing, request processing, session management, rendering templates, redirects, flash messages, and HTTP responses.

Database:
- SQLite is used as the local relational database.
- The database file is tas_reports.db.
- SQLite is accessed using Python's built-in sqlite3 module.

Frontend:
- HTML templates are stored in the templates folder.
- CSS styling is stored in static/css/style.css.
- JavaScript behavior is stored in static/js/app.js.
- Jinja2 template rendering is used through Flask's render_template function.

Resume parsing:
- pypdf or PyPDF2 is used for PDF text extraction.
- python-docx is used for DOCX text extraction.
- Plain TXT files are read directly.
- pytesseract and pdf2image are optionally supported for OCR on scanned PDFs.

Data processing:
- pandas is used to load and update job_data.csv.
- numpy is used for score clipping and numeric handling.
- spaCy is optionally used for NLP entity extraction. If the configured spaCy model is not available, the app falls back to a blank spaCy model or rule-based logic.

Security and utilities:
- Werkzeug secure_filename is used to sanitize uploaded filenames.
- Werkzeug password hashing is used to hash and verify user passwords.
- secrets is used for session secret generation and CSRF token generation.
- UUIDs are used to generate safe temporary upload filenames.

Testing:
- pytest is used for automated tests.
- Tests are located in tests/test_app.py.

4. APPLICATION FLOW
-------------------
Typical resume scan flow:

1. User opens the home page at /.
2. User uploads a resume file and optionally enters a target job title and job description.
3. The app validates the file extension using allowed_file().
4. The file is saved temporarily using save_upload_temporarily().
5. The app checks whether the file content matches the file extension using file_signature_matches().
6. Text is extracted using extract_resume_text().
7. The temporary uploaded file is deleted after processing.
8. The extracted text is normalized using normalize_text().
9. The resume profile is built using extract_resume_profile().
10. The target role match is built using build_target_job_match().
11. Criteria scores are calculated using build_criteria_scores().
12. Score explanations are generated using build_score_details().
13. The total role-adjusted score is calculated using calculate_role_adjusted_score().
14. Recommended jobs are found using find_applicable_jobs().
15. Suggestions are generated using build_resume_suggestions().
16. Analysis steps are generated using build_analysis_steps().
17. Bullet rewrites are generated using rewrite_resume_bullets().
18. The report is saved to SQLite using save_report().
19. The result page is rendered using templates/result.html.


5. DATABASE USAGE
-----------------
Database file:
- tas_reports.db

Database engine:
- SQLite

Database initialization:
- init_db() creates required tables if they do not already exist.
- init_db() is called when the app starts from app.py.
- Tests also call init_db() with a temporary test database.

Tables used:

1. users
Columns:
- id: integer primary key
- username: unique username
- password_hash: hashed password

Purpose:
- Stores user accounts.
- Used for registration, login, current user lookup, access control, and report ownership.

Where used:
- create_default_user() inserts a default admin user if no users exist.
- create_user() creates a new user account.
- get_user() retrieves a user by username.
- get_current_user() reads the username from the Flask session and returns user details.
- login() checks credentials using the users table.
- register() creates users after validation.
- admin_required() checks whether the current user is the configured admin.
- get_accessible_user_ids() uses users to decide which reports a user can access.

2. reports
Columns:
- id: integer primary key
- user_id: owner of the report; can be null for anonymous scans
- filename: original uploaded filename
- uploaded_at: UTC timestamp when report was saved
- score: total TAS score
- matched_jobs: JSON text containing recommended jobs and match information
- criteria: JSON text containing category-wise scores
- suggestions: JSON text containing improvement suggestions
- analysis: JSON text containing extracted profile details, score explanations, scoring weights, analysis steps, and bullet rewrites

Purpose:
- Stores resume scan results for history and report export.
- It allows logged-in users to revisit previous scans.

Where used:
- save_report() inserts a new resume scan result.
- get_latest_score() retrieves the user's most recent score for comparison.
- history() loads recent reports and builds dashboard statistics.
- get_report_payload() retrieves a full report for detail view and PDF export.
- normalize_report_payload() converts stored JSON text back into Python data structures.
- report_lines() converts report data into text lines for PDF export.
- export_report_pdf() generates a downloadable PDF from stored report data.
- admin() counts reports for dashboard summary.
- ensure_report_columns() and ensure_reports_user_scope() migrate older report tables by adding missing columns.
- assign_legacy_reports_to_default_user() assigns old reports without user_id to a default user.

3. contacts
Columns:
- id: integer primary key
- name: contact person's name
- email: contact email
- subject: optional subject
- message: submitted message
- created_at: UTC timestamp

Purpose:
- Stores contact/support messages submitted from the contact page.

Where used:
- save_contact_submission() inserts contact form data.
- contact() validates form input and saves submissions.
- admin() displays the latest contact submissions.

Why JSON is stored in SQLite:
- matched_jobs, criteria, suggestions, and analysis contain nested structures such as lists and dictionaries.
- SQLite does not require a complex schema for these flexible report details.
- The app serializes these values using json.dumps() before saving and restores them using json.loads() through parse_report_payload().

Privacy and runtime behavior:
- Uploaded resumes are temporary and deleted after text extraction.
- The original resume file is not permanently stored.
- The database stores extracted analysis results, not the uploaded file itself.
- tas_reports.db is treated as a runtime file because it may contain private user data.

7. SCORING SYSTEM
-----------------
The resume is scored across ten categories:

1. Contact Information
Checks whether the resume has email, phone, LinkedIn, GitHub, or portfolio signals.

2. Professional Summary
Checks for summary or objective-style indicators.

3. Work Experience
Checks for bullet points, action verbs, years of experience, and measurable impact.

4. Education
Checks for education-related terms such as bachelor, master, degree, university, college, diploma, and related signals.

5. Skills
Uses SKILL_SYNONYMS to detect technical and soft skills, including Python, SQL, pandas, Excel, Flask, JavaScript, machine learning, API development, Git, HTML, CSS, communication, leadership, and testing.

6. TAS Optimization
Checks resume length, bullet usage, and whether the format is likely readable by a TAS.

7. Consistency
Checks consistency signals such as date formatting and bullet usage.

8. Proofreading
Checks common typos such as "teh", "recieve", "acheive", "managment", and similar mistakes.

9. File Format
Scores based on the uploaded extension. PDF, DOCX, and TXT are supported.

10. Relevance
Compares resume text with the target job title and description if provided.

Total score:
- calculate_weighted_score() uses the default SCORING_WEIGHTS.
- calculate_role_adjusted_score() adjusts category weights based on target role.
- scoring_weights_for_role() applies role-specific overrides for roles like Data Analyst, Software Engineer, Developer, Product Manager, and Technical Writer.

Example:
- For a Data Analyst role, Skills and Relevance receive more weight because technical and role-specific keyword alignment are especially important.


8. JOB MATCHING SYSTEM
----------------------
The app uses two types of job matching:

1. Target role matching:
- If the user enters a job title or job description, build_target_job_match() builds a match object.
- It extracts keywords from the title and description.
- It expands known titles using ROLE_KEYWORD_HINTS.
- It identifies matched and missing keywords.
- It calculates a match score.

2. Catalog role matching:
- find_applicable_jobs() reads job_data.csv using load_job_data().
- Each catalog job has a title, keywords, and description.
- The app compares resume text and extracted profile data with each job.
- It returns ranked job recommendations.

Similarity methods:
- tokenize_for_similarity() tokenizes text into useful words.
- build_weighted_terms() gives stronger weight to important terms.
- weighted_cosine_similarity() compares weighted term counters.
- cosine_similarity() provides a simpler token-list comparison.
- keyword_found_in_text() checks whether a keyword appears in normalized text.


9. RESUME IMPROVEMENT FEATURES
------------------------------
Suggestions:
- build_resume_suggestions() looks at weak categories, missing keywords, weak verbs, and profile gaps.
- It returns practical suggestions for improving the resume.

Analysis steps:
- build_analysis_steps() produces a short explanation of what the system did:
  1. Parsed Resume
  2. Checked TAS Signals
  3. Compared Target Role
  4. Generated Fixes

Bullet rewrites:
- extract_improvable_bullets() finds bullets that are weak or lack measurable impact.
- rewrite_resume_bullets() suggests stronger rewritten versions using action-oriented language.

PDF export:
- report_lines() formats saved report data as text.
- build_simple_pdf() creates a minimal PDF file directly in Python.
- export_report_pdf() returns that PDF to the browser.


10. SECURITY AND VALIDATION
---------------------------
File upload security:
- allowed_file() restricts uploads to PDF, DOCX, and TXT.
- secure_filename() sanitizes the original filename.
- save_upload_temporarily() saves the file under a UUID name.
- file_signature_matches() checks the file header to reject mismatched extensions.
- Uploaded files are deleted after analysis.
- MAX_CONTENT_LENGTH limits uploads to 10 MB.

Authentication:
- Passwords are stored as hashes using Werkzeug.
- check_password_hash() validates login credentials.
- Flask sessions store the logged-in username.

CSRF protection:
- csrf_token() creates a token stored in the session.
- inject_csrf_token() makes the token available to templates.
- protect_from_csrf() validates POST, PUT, PATCH, and DELETE requests.
- CSRF can be disabled for tests using DISABLE_CSRF=1 or test config.

Rate limiting:
- rate_limit() limits repeated POST requests by route and client IP.
- Used on login, register, contact, index, and admin routes.
- RATE_LIMITS is an in-memory dictionary.

Access control:
- login_required() protects history and export pages.
- admin_required() protects the admin dashboard.
- get_accessible_user_ids() ensures users only access allowed reports.

Input validation:
- Email format is validated using EMAIL_PATTERN.
- Contact form and target description length are limited.
- Admin catalog fields are length-limited.

14. CURRENT LIMITATIONS AND FUTURE IMPROVEMENTS
-----------------------------------------------
Current limitations:
- Scoring is rule-based and keyword-based, so it should be treated as guidance rather than a real TAS guarantee.
- SQLite is suitable for a local/demo project, but a production app should use a stronger database such as PostgreSQL.
- In-memory rate limiting works for a single process, but production should use Redis or another shared store.
- OCR depends on external native tools, Tesseract and Poppler.
- The job catalog is small and stored in CSV.

Future improvements:
- Use embeddings or a stronger semantic matching model for better resume/job comparison.
- Add richer admin analytics.
- Add email notifications for contact form submissions.
- Move job catalog management fully into the database.
- Add password reset and account management.
- Add more export formats.
- Deploy with PostgreSQL, environment-managed secrets, and production logging.
